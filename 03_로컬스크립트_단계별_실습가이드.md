## STEP 1 — `explore.py` : 설치 없이 온톨로지 구조 파악하기

### 목적
> 노드와 엣지가 실제로 어떻게 연결되어 있는지  
> 9가지 시맨틱 스페이스가 실제 데이터에서 어떻게 보이는지 눈으로 확인

### 실행
```powershell
cd "c:\Users\HYKP\Vscode\04 Study\Ontology"
python explore.py
```

### 메뉴별 학습 포인트

| 메뉴 | 무엇을 배우나 |
|------|-------------|
| **1. 전체 통계** | 9 스페이스별 노드 분포를 막대그래프로 확인. 내 온톨로지가 균형 잡혀있는지 파악하는 습관 |
| **2. 스페이스별 노드 목록** | `Claim` 입력 → "Python은 ML 표준" 같은 주장 노드가 왜 따로 존재하는지 이해 |
| **3. 노드 연결 관계 조회** | `subj_python` 입력 → Python이 무엇과 연결되었나. 온톨로지의 핵심인 "관계의 망" 체험 |
| **4. 키워드 검색** | `ML` 검색 → 관련 노드·문서가 한번에. RAG와 달리 구조를 따라 검색됨 |
| **5. Claim → Evidence 추적** | 주장의 근거를 출처까지 추적. 온톨로지가 일반 문서와 다른 핵심 가치 |
| **6. Intensity 랭킹** | 수치화된 친화도 점수. 지식의 "강도"를 표현하는 방법 |
| **7. 전체 그래프 출력** | 모든 관계를 타입별로 그룹화. 내 온톨로지의 어휘집(relation vocabulary) 파악 |

### 실습 체크리스트
```
□ "Subject" 스페이스 → 5개 언어 노드 확인
□ "subj_python" 연결 관계 조회 → 나가는/들어오는 관계 구분 이해
□ "ML" 키워드 검색 → 노드와 문서가 함께 검색되는 구조 확인
□ Claim 메뉴 → "Python은 AI/ML 표준" 주장의 근거 출처까지 추적
□ Intensity 랭킹 → 0.99(JS 웹 지배력) vs 0.95(Python-ML 친화도) 비교
```

### `queries.cypher` 단계별 실습

Neo4j Browser (`http://localhost:7474`) 좌측 에디터에 쿼리를 붙여넣고 **Ctrl+Enter** 로 실행.

#### LEVEL 1 — 기본 확인 (데이터가 잘 들어갔나?)
```cypher
-- 노드 수 확인
MATCH (n:OntologyNode) RETURN count(n) AS 총_노드수;

-- 스페이스별 분포 (9가지가 다 보여야 함)
MATCH (n:OntologyNode)
RETURN n.space AS 스페이스, count(n) AS 노드수
ORDER BY 노드수 DESC;
```

#### LEVEL 2 — 특정 노드 탐색 (핵심 개념)
```cypher
-- Python 노드와 직접 연결된 모든 것 (시각화 추천)
MATCH (n:OntologyNode {id: 'subj_python'})-[r]-(m)
RETURN n, r, m;
```
> 💡 결과 탭에서 **"Graph" 뷰**로 전환하면 노드-엣지가 시각적으로 보임

```cypher
-- ML 개념과 관련된 2단계까지
MATCH path = (n:OntologyNode {id: 'con_ml'})-[*1..2]-(m)
RETURN path;
```

#### LEVEL 3 — 시맨틱 스페이스별 탐색
```cypher
-- Claim → Evidence 연결 (주장의 근거 추적)
MATCH (c:OntologyNode {space: 'Claim'})-[:SUPPORTED_BY]->(e:OntologyNode {space: 'Evidence'})
RETURN c.label AS 주장, e.label AS 근거, e.source AS 출처;
```

#### LEVEL 4 — 분석 쿼리 (가장 중요한 노드는?)
```cypher
-- 연결 수 기준 핵심 노드 찾기 (중심성 분석)
MATCH (n:OntologyNode)-[r]-()
RETURN n.label AS 노드, n.space AS 스페이스, count(r) AS 연결수
ORDER BY 연결수 DESC LIMIT 10;
```

#### LEVEL 5 — 직접 데이터 추가 (나만의 노드 넣기)
```cypher
-- Go 언어 노드 추가
CREATE (n:OntologyNode {
  id: 'subj_go', label: 'Go', space: 'Subject',
  created_year: 2009, creator: 'Google',
  description: '동시성 처리에 강한 정적 타입 컴파일 언어'
});

-- 관계 연결
MATCH (go:OntologyNode {id: 'subj_go'})
MATCH (web:OntologyNode {id: 'con_web_dev'})
CREATE (go)-[:EXCELS_AT {strength: 0.85}]->(web);
```

### Cypher 실습 체크리스트
```
□ LEVEL 1 — 노드 34개, 관계 48개 확인
□ LEVEL 2 — subj_python Graph 뷰로 시각화
□ LEVEL 2 — con_ml 2단계 경로 탐색
□ LEVEL 3 — Claim → Evidence 추적 (텍스트 뷰로 확인)
□ LEVEL 4 — 연결수 TOP 10에서 Python과 con_web_dev 상위 확인
□ LEVEL 5 — Go 언어 노드 직접 추가 후 확인
```

## STEP 3 — `ai_context.py` : 온톨로지를 AI의 지식으로 주입하기

### 목적
> 구축한 온톨로지를 OpenCrab·MCP 없이 AI에게 직접 먹이는 방법  
> 일반 질문과 온톨로지 기반 질문의 답변 품질 차이를 비교

### 핵심 개념: 왜 이게 의미있나?

```
일반 Claude 질문:
  Q: "Python이 ML에 좋은 이유는?"
  A: 학습 데이터 기반의 일반론적 답변

온톨로지 컨텍스트 주입 후:
  Q: "Python이 ML에 좋은 이유는?"
  A: 이 온톨로지의 노드(TensorFlow, PyTorch, NumPy)와
     Evidence(TIOBE 2024, GitHub Octoverse 2023)를
     구체적으로 인용한 구조화된 답변
```

### 실행

```powershell
python ai_context.py
```

```
1. 오프라인 모드  → 컨텍스트 텍스트 생성 (붙여넣기용, API 키 불필요)
2. API 모드       → Claude에게 직접 질의 (ANTHROPIC_API_KEY 필요)
```

### 오프라인 모드 실습 (API 키 없어도 가능)

```
선택: 1
질문: Python이 ML에 강한 이유는?
→ generated_context.txt 생성됨
→ 이 파일 내용을 claude.ai 채팅창에 붙여넣고 전송
→ 온톨로지 기반으로 답변하는지 확인
```

**좋은 테스트 질문 예시:**
```
"Python이 ML에 강한 이유를 Evidence 기반으로 설명해줘"
"TypeScript vs JavaScript — 어떤 Claim 노드가 있어?"
"웹 개발 관련 노드들을 모두 찾아서 관계를 정리해줘"
"Intensity 점수가 가장 높은 항목은 뭐고 왜 그런가?"
```

## 내 도메인으로 확장하기 — 커스텀 온톨로지 만드는 법

STEP 1~3을 이해했다면, 이제 본인 도메인으로 적용할 차례입니다.

### 설계 단계 — 9가지 스페이스에 내 도메인 매핑하기

| 스페이스 | 프로그래밍 언어 예시 | 내 도메인 예시 (빈칸) |
|---------|------------------|------------------|
| Subject | Python, JavaScript | |
| Resource | TensorFlow, React | |
| Evidence | TIOBE 2024, SO 설문 | |
| Concept | ML, 웹개발, OOP | |
| Result | Python GitHub 1위 | |
| Claim | "Python은 ML 표준" | |
| Community | PyPI, npm | |
| Policy | PEP 8, ECMAScript | |
| Intensity | Python-ML 친화도 0.95 | |

### 데이터 작성 규칙 (nodes.jsonl)

```jsonl
{"id": "고유ID", "label": "표시이름", "type": "스페이스명", "space": "스페이스명",
 "properties": {"description": "설명", "기타속성": "값"}}
```

**id 작명 규칙:** `{스페이스약자}_{이름}` 형태 권장
```
subj_python  → Subject의 Python
res_django   → Resource의 Django
evd_tiobe    → Evidence의 TIOBE
con_ml       → Concept의 ML
```

### 데이터 작성 규칙 (edges.jsonl)

```jsonl
{"source": "출발노드ID", "target": "도착노드ID", "relation": "관계명",
 "properties": {"strength": 0.8, "기타": "값"}}
```

**relation 작명 규칙:** 대문자 + 언더스코어
```
EXCELS_AT       → ~에 특화됨
BUILT_WITH      → ~로 만들어짐
SUPPORTED_BY    → ~에 의해 지지됨
GOVERNED_BY     → ~의 규칙을 따름
HAS_ECOSYSTEM   → 생태계를 보유
```

### 추천 첫 번째 커스텀 주제

| 난이도 | 주제 | 이유 |
|--------|------|------|
| ⭐ 쉬움 | 내가 공부한 기술 스택 | 이미 알고 있는 내용이라 노드 설계 쉬움 |
| ⭐⭐ 보통 | 책/강의 한 챕터 정리 | 범위가 명확해서 설계하기 좋음 |
| ⭐⭐⭐ 어려움 | 회사 업무 도메인 | 추상화와 관계 정의가 필요 |
